3) Critical Architecture: Tenant Creation Must Work
====================================================================

Definition of done for tenant provisioning (P0):
- Create tenant record in central DB.
- Create tenant MySQL database/schema.
- Run tenant migrations.
- Seed minimum tenant data (roles, default settings, baseline package limits).
- Create owner admin user for tenant.
- Return tenant access URL/domain and initial credentials delivery flow.

Suggested provisioning flow:
1. API receives create-tenant request (validated).
2. Persist central tenant row with status = provisioning.
3. Dispatch provisioning job chain (idempotent).
4. Create MySQL DB and apply tenant connection config.
5. Run tenant migrations + seeders.
6. Create default role set and owner user.
7. Mark tenant status = active and emit domain-ready event.
8. Send notification/webhook for external systems.

Failure and recovery requirements:
- Idempotency key for create-tenant API.
- Retry-safe provisioning jobs (no duplicate tenant DB creation).
- Compensating action on partial failure (status = failed, support action payload).
- Structured logs for each provisioning stage.
- Horizon tags include tenant_id and provisioning_run_id.

4) Tenant UI/UX Direction (Modern, Dynamic, Purposeful)
====================================================================

UX principles:
- Fast time-to-insight on dashboard landing.
- Motion used to explain state changes (not decorative noise).
- Action-first cards: each metric card has one primary follow-up action.
- Responsive layout for office desktop and on-site mobile operations.

UI capabilities to include:
- Chart layer for revenue, bookings, payment aging, and staffing workload trends.
- Live tiles (pending approvals, due payments, active events today).
- Filterable timelines for event progress.
- Slide-over detail panels to avoid page reload churn.

Recommended frontend stack options (choose one in app repo):
- Livewire + Alpine + ApexCharts/Chart.js for rapid Laravel-native delivery.
- Inertia + Vue/React + ECharts/Recharts for richer SPA interactions.

Purposeful animation guidelines:
- 120-220ms transitions for panel open/close and KPI updates.
- Staggered load-in only on first dashboard render.
- Realtime count transitions (delta flashes) for booking/payment updates.
- Respect reduced-motion settings for accessibility.

5) Role-specific Tenant Dashboards (Critical)
====================================================================

| Role | Core Capabilities | Key Widgets | Primary Actions |
|---|---|---|---|
| Admin | Full access; manage users; configure packages/pricing; financial reports; system settings | MRR/revenue trend, plan utilization, unpaid balance aging, user activity feed | Create user, change role, edit package limits, run financial exports, update tenant settings |
| Manager | Approve bookings; monitor events; view reports | Booking approval queue, event calendar, staff load, event completion funnel | Approve/reject booking, assign staff, adjust event schedule, generate ops report |
| Staff | View assigned events; update event status | My assignments today, task checklist, event status timeline | Mark stage complete, add event notes, flag issue/blocker |
| Cashier | Record payments; monitor balances; generate receipts | Due today, outstanding invoices, payment method mix | Record payment, reconcile balance, generate/send receipt |

Access control requirements:
- Enforce role permissions with policy + permission middleware.
- Restrict data queries to tenant boundary and role scope.
- Audit sensitive actions (pricing changes, refunds, role changes).

6) API Inventory (Implementation Starter)
====================================================================

Conventions:
- Versioned prefix: /api/v1
- Auth: Sanctum bearer tokens for API clients
- All tenant endpoints require tenant context resolution and authorization

| Domain | Endpoint Examples | Purpose |
|---|---|---|
| Auth | POST /auth/login, POST /auth/logout, POST /auth/token, GET /auth/me | Session/token auth, current user profile |
| Tenant Provisioning | POST /tenants, GET /tenants/{id}, POST /tenants/{id}/retry-provision, GET /tenants/{id}/provision-status | Create tenant, status checks, controlled retry |
| Dashboard Stats | GET /dashboard/admin, GET /dashboard/manager, GET /dashboard/staff, GET /dashboard/cashier | Role-specific KPI payloads |
| Users | GET /users, POST /users, PATCH /users/{id}, PATCH /users/{id}/role, DELETE /users/{id} | User lifecycle and role assignment |
| Bookings/Events | GET /events, POST /events, PATCH /events/{id}, POST /events/{id}/approve, POST /events/{id}/assign-staff | Booking approval and event operations |
| Packages | GET /packages, POST /packages, PATCH /packages/{id}, PATCH /packages/{id}/pricing | Package and pricing management |
| Payments | GET /payments, POST /payments, GET /payments/balances, POST /payments/{id}/reconcile | Payment recording and balance tracking |
| Reports | GET /reports/financial, GET /reports/events, GET /reports/staffing | Downloadable and on-screen reporting |
| Receipts | POST /receipts, GET /receipts/{id}, POST /receipts/{id}/send | Receipt generation and delivery |
| Notifications | GET /notifications, PATCH /notifications/{id}/read, POST /notifications/test | In-app and test dispatch |
| Webhooks | POST /webhooks/payments, POST /webhooks/external-bookings | External event ingestion |
| Analytics | GET /analytics/revenue-trend, GET /analytics/booking-funnel, GET /analytics/payment-aging | Chart-ready trend endpoints |

API quality requirements:
- Form request validation on all write endpoints.
- Consistent response envelope and error codes.
- Idempotency on provisioning and payment-critical endpoints.
- OpenAPI spec maintained from phase 1 onward.

8) Acceptance Criteria (by capability)
====================================================================

Tenant provisioning (P0):
- Given a valid create request, tenant reaches active state within defined SLO.
- Tenant DB exists in MySQL and passes migration checksum.
- Owner admin can authenticate and access tenant domain.
- Failed provisioning is observable, retryable, and does not duplicate resources.

Role dashboards:
- Each role sees only authorized widgets and actions.
- Dashboard loads return chart-ready data within performance budget.
- Realtime updates apply without full page reload.

Payments and receipts:
- Cashier can post payment and see updated balance immediately.
- Receipt generation produces a stored artifact and retrievable record.
- Financial report totals reconcile with ledger entries.

Platform quality:
- Queue failure visibility is available in Horizon.
- Pulse displays key app and queue performance signals.
- Feature flags can disable non-critical modules without breaking core flows.
